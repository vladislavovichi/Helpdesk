# Деплой И Обновление

## Что Считать Безопасным Порядком

1. Проверить актуальность `.env` и убедиться, что внутренние токены не дрейфовали.
2. Поднять или обновить стек командой `make up`.
3. Дождаться, пока `make health` перестанет показывать `starting`.
4. Прогнать `make smoke`.
5. Только после этого считать релиз пригодным к приёмке.

Если нужен полностью production-like запуск без dev bind-mount, используйте только базовый compose-файл:

```bash
docker compose -f ops/docker/compose.yml up --build -d
```

Обычный `make up` по умолчанию добавляет `ops/docker/compose.dev.yml`, чтобы локальная разработка оставалась удобной.

## Что Делает Стек При Старте

- `ai-service` стартует без миграций;
- `backend` на старте выполняет Alembic `upgrade head`;
- `bot` миграции не запускает;
- если `backend` не может дойти до БД, Redis или `ai-service`, он завершится с ошибкой;
- если `bot` не может дойти до `backend`, он не остаётся в ложном ready-состоянии.

## Команды Для Оператора

```bash
make up
make ps
make logs
make logs-backend
make logs-ai
make smoke
make down
```

Для ручного запуска миграций внутри Docker-стека:

```bash
make migrate-stack
```

## Проверка После Обновления

Минимум после каждого обновления:

```bash
make health
make smoke
```

Хороший сигнал:

- все сервисы healthy;
- smoke-check доходит до backend gRPC;
- smoke-check подтверждает доступность `postgres`, `redis`, `ai-service`;
- bot runtime diagnostics не падают.

## Осторожный Rollback

Rollback здесь не автоматизирован. Перед откатом нужно отдельно понять:

- применялись ли новые миграции;
- совместим ли старый код с уже обновлённой схемой БД;
- не менялись ли внутренние auth-токены;
- не полагается ли новый код на уже записанные данные нового формата.

Если релиз не прошёл smoke-check, не спешите просто откатывать контейнеры. Сначала посмотрите логи `backend`, потому что именно он выполняет миграции и обычно даёт первичную правду о причине неуспеха.
